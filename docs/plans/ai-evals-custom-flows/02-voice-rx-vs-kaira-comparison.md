# Voice RX vs Kaira Bot: Evaluation Flow Comparison

## Side-by-Side: Standard (Built-in) Evaluations

| Dimension | Voice RX | Kaira Bot Batch | Kaira Bot Adversarial |
|-----------|----------|-----------------|----------------------|
| **eval_type** | `full_evaluation` | `batch_thread` | `batch_adversarial` |
| **Input source** | Single listing (audio + transcript/API response) | CSV data file with thread conversations | Live Kaira API |
| **Pipeline shape** | Sequential 3-step: transcribe → normalize → critique | Fan-out: N threads × 3 evaluators | Generate → converse → judge |
| **Step controllers** | `FlowConfig` (frozen dataclass) | Boolean toggles (`evaluate_intent`, etc.) | `AdversarialConfig` (Pydantic model from DB) |
| **Prompt management** | Transcription: DB defaults. Critique: hardcoded constants | All hardcoded in evaluator classes | Generation: dynamic from config. Judging: hardcoded |
| **Schema management** | Transcription: DB defaults. Critique: hardcoded constants | All hardcoded in evaluator classes | Generation: dynamic from config. Judging: hardcoded |
| **Variable resolution** | `prompt_resolver.py` for transcription prompt | Internal `_format_transcript()` per evaluator | Dynamic prompt builders from config categories |
| **Server-side logic** | Comparison table building, server-computed statistics | Rule injection via `rule_catalog.py` | Config-driven category/rule filtering |
| **Result storage** | `EvalRun.result` (single structured dict with judgeOutput, critique, etc.) | `ThreadEvaluation.result` per thread (nested evaluator outputs) | `AdversarialEvaluation.result` per test case |
| **Summary computation** | `_build_summary()` — accuracy from segment/field counts | Aggregated across threads: avg_intent, verdict distributions | Verdict/category distributions |
| **Config snapshot** | `EvalRun.config` (prompts, schemas, models, prerequisites) | `EvalRun.batch_metadata` (toggles, evaluator IDs) | `EvalRun.batch_metadata.adversarial_config` |
| **Parallelism** | None (sequential steps) | `parallel_engine.run_parallel()` for threads | `parallel_engine.run_parallel()` for test cases |
| **Custom eval support** | Separate job type (`evaluate-custom`) | Inline — custom evaluators run alongside built-ins per thread | None |

## Standard Pipeline: What Makes It "Standard"

### Voice RX Standard Contract

The Voice RX standard pipeline has a **strict contract** that must be preserved:

1. **Two-call pipeline**: Call 1 = Transcription (with audio), Call 2 = Critique (text-only)
2. **Transcription prompt/schema**: Loaded from `prompts`/`schemas` DB tables with `is_default=True`, seeded by `seed_defaults.py`
3. **Critique prompt/schema**: Hardcoded in `evaluation_constants.py` — `UPLOAD_EVALUATION_PROMPT`, `UPLOAD_EVALUATION_SCHEMA`, `API_EVALUATION_PROMPT`, `API_EVALUATION_SCHEMA`
4. **Comparison table**: Built server-side by `comparison_builder.py` (API flow) or inline segment comparison (upload flow) — never generated by LLM
5. **Statistics**: Always computed server-side from known segment/field counts — never trust LLM counts
6. **FlowConfig**: Immutable frozen dataclass created once at start, controls all step behavior
7. **Normalization**: Optional step, non-fatal failure (logs warning, continues without)

### Kaira Bot Standard Contract

The Kaira batch pipeline has a looser but still fixed contract:

1. **Three built-in evaluators**: `IntentEvaluator`, `CorrectnessEvaluator`, `EfficiencyEvaluator`
2. **Each evaluator** has its own hardcoded system prompt, evaluation prompt template, and JSON schema
3. **Rule injection**: `rule_catalog.py` provides production prompt rules that are injected into correctness and efficiency prompts
4. **Per-thread isolation**: Each thread is evaluated independently; results saved to `thread_evaluations`
5. **Partial failure tolerance**: Individual evaluator failures don't abort the thread; results stored with `failed_evaluators` metadata

### Kaira Adversarial Standard Contract

1. **Config-driven categories**: `AdversarialConfig` (Pydantic model) loaded from settings table
2. **Two-phase**: Generate test cases (LLM) → Run conversations (live API) → Judge transcripts (LLM)
3. **Rule compliance**: Production rules from config injected into judge prompt
4. **Per-case isolation**: Each test case is independent; failures saved with error details

## Custom Pipeline: What Makes It "Custom"

### Current Custom Evaluator Model

The `Evaluator` entity represents a user-defined evaluation:

```python
class Evaluator:
    id: UUID
    app_id: str                   # "voice-rx" or "kaira-bot"
    listing_id: UUID | None       # Scoped to a listing, or None for global
    name: str                     # User-given name
    prompt: str                   # User-written prompt with {{variables}}
    model_id: str | None          # Override LLM model
    output_schema: list[dict]     # Field-based output definition
    is_global: bool               # Available across all listings/sessions
    show_in_header: bool          # Display in evaluation header cards
    forked_from: UUID | None      # Lineage tracking
```

### How Custom Evals Execute

**Single entity (Voice RX or Kaira):**
1. `custom_evaluator_runner.py` loads the Evaluator + entity (Listing or ChatSession)
2. `prompt_resolver.py` resolves `{{variables}}` from entity data
3. `schema_generator.py` converts `output_schema` fields to JSON Schema
4. Single LLM call: `generate_json()` or `generate_with_audio()`
5. Result stored in `EvalRun` with `eval_type='custom'`

**Batch on single entity (Voice RX or Kaira):**
1. `voice_rx_batch_custom_runner.py` runs N evaluators on one entity
2. Each creates its own `EvalRun` via `custom_evaluator_runner`
3. Runs in parallel by default

**Inline in Kaira batch:**
1. `batch_runner.py` loads custom evaluators by ID (`custom_evaluator_ids` param)
2. For each thread, runs built-in evaluators + custom evaluators
3. Custom results stored in `ThreadEvaluation.result.custom_evaluations`
4. Summary aggregated in `EvalRun.summary.custom_evaluations`

## Key Structural Differences

### 1. Prompt Ownership

```
Voice RX Standard:
  Transcription → DB (editable via Prompts UI, but seeded defaults)
  Critique      → Code (evaluation_constants.py, never user-editable)

Kaira Batch Standard:
  Intent      → Code (intent_evaluator.py INTENT_JSON_SCHEMA)
  Correctness → Code (correctness_evaluator.py CORRECTNESS_JUDGE_PROMPT)
  Efficiency  → Code (efficiency_evaluator.py EFFICIENCY_JUDGE_PROMPT)

Custom:
  All prompts → DB (evaluators.prompt, fully user-editable)
```

**Analysis**: Voice RX has a hybrid model (some DB, some code). Kaira is fully code-driven. Custom is fully DB-driven. The inconsistency means:
- Voice RX transcription prompt is theoretically editable in the Prompts UI, but the standard pipeline always loads `is_default=True`
- Kaira built-in evaluators have no DB representation at all — they're pure code
- Custom evaluators are fully flexible but have no type safety on their prompts

### 2. Schema Ownership

Same pattern as prompts:
- Voice RX transcription schema is in DB, critique schema is hardcoded
- Kaira evaluator schemas are hardcoded per evaluator class
- Custom evaluator schemas are field-based definitions in DB

### 3. Variable Resolution

Voice RX and custom evaluators share `prompt_resolver.py`, but Kaira built-in evaluators format their own prompts internally:

```python
# IntentEvaluator — formats history + query text directly
eval_prompt = f"""{history_context}
User Query: "{message.query_text}"
Classify this query..."""

# CorrectnessEvaluator — formats user input + bot response + rules
eval_prompt = (
    f"### Current turn\n**User input:** {message.query_text}\n"
    f"**Bot response:**\n{message.final_response_message}\n"
    f"{rules_block}\n"
)

# EfficiencyEvaluator — formats full transcript + rules
eval_prompt = (
    f"### Conversation thread ({thread.message_count} turns)\n"
    f"{transcript}\n{rules_block}\n"
)
```

This means Kaira built-in evaluators have **no shared variable resolution** — each evaluator builds its own prompt from raw data.

### 4. Result Shape

```python
# Voice RX standard result:
{
    "judgeOutput": {"transcript": ..., "segments": [...]},
    "normalizedOriginal": {...},
    "critique": {"flowType": ..., "statistics": {...}, "segments": [...]},
    "status": "completed",
    "flowType": "upload"
}

# Kaira batch per-thread result (ThreadEvaluation.result):
{
    "thread": {/* serialized ConversationThread */},
    "intent_evaluations": [{/* IntentEvaluation */}],
    "correctness_evaluations": [{/* CorrectnessEvaluation */}],
    "efficiency_evaluation": {/* EfficiencyEvaluation */},
    "success_status": true,
    "custom_evaluations": {"<evaluator_id>": {"status": ..., "output": {...}}}
}

# Custom evaluator result:
{
    "output": {/* user-defined shape from output_schema */},
    "rawRequest": "resolved prompt text",
    "rawResponse": "raw LLM response"
}
```

### 5. Seeded Evaluators

**Voice RX**: No seeded evaluators. The standard pipeline is code, not an Evaluator entity.

**Kaira Bot**: 4 seeded global evaluators in `seed_defaults.py`:
- Chat Quality Analysis
- Health Accuracy Checker
- Empathy Assessment
- Risk Detection

These are `Evaluator` rows in the DB with `is_global=True`, but they're used as **custom evaluators**, not as the standard pipeline. The standard pipeline (intent/correctness/efficiency) has no DB representation.

**Implication**: The Kaira "standard" evaluators (intent, correctness, efficiency) and the seeded "custom" evaluators (Chat Quality, Health Accuracy, Empathy, Risk Detection) are **conceptually different things** that happen to live in different code paths:
- Standard: hardcoded classes, run by `batch_runner.py`
- Seeded custom: DB Evaluator rows, run by `custom_evaluator_runner.py` or inline in batch

## What This Means for the Separation

1. **Voice RX standard pipeline is already well-isolated** — it's a single runner with clear boundaries. The "standard" label comes from the hardcoded critique prompts/schemas and server-side comparison logic.

2. **Kaira standard pipeline is well-isolated in code** but lacks formal representation — the built-in evaluators are classes, not entities. There's no Evaluator row in the DB for "Intent Accuracy" or "Correctness."

3. **Custom evaluators are well-separated** — they have their own `eval_type`, their own runner, and their own DB model. The overlap is in batch_runner where customs run alongside built-ins.

4. **The variable system needs formalization** — `prompt_resolver.py` is the closest thing to a variable registry, but it's a bag of if/elif statements with no metadata, no per-app scoping, and no documentation for users.

5. **Frontend has a hardcoded duplicate** — `src/services/templates/variableRegistry.ts` (395 lines) maintains a parallel copy of variable metadata with its own `TEMPLATE_VARIABLES`, `getVariablesForApp()`, `validatePromptVariables()`, and step-aware filtering. This is consumed by `VariablePickerPopover.tsx`. The two registries (backend resolver + frontend registry) can drift out of sync. Solution: backend becomes single source of truth, frontend fetches via API.

## Score Engine Analysis

The custom eval pipeline's **metrics engine** is the chain: `output_schema` → `_extract_scores()` → `summary` → `OutputFieldRenderer`.

### Current `_extract_scores()` Logic (in `custom_evaluator_runner.py`)

1. Find `isMainMetric` field → extract `overall_score`
2. Guess `max_score` from `thresholds.green` or default to `100`
3. Find "reasoning" by substring matching field keys (`"reason"`, `"explanation"`, `"comment"`)
4. Build `breakdown` from all non-hidden fields

### Problems

| Issue | Impact |
|-------|--------|
| Substring matching for reasoning is fragile | Won't find "justification", "notes", "rationale" |
| `max_score = 100` default is arbitrary | Misleading for percentage (0-1) or 10-point scales |
| Only one `isMainMetric` supported | Users with multiple important metrics lose visibility |
| No `enum` field type | LLM can return any string for verdicts; `VerdictBadge` guesses via `str === str.toUpperCase()` heuristic |
| No aggregation across runs | Running same evaluator on 50 listings shows no trend/average |

### Solution

1. Add explicit `role` field (`metric` | `reasoning` | `detail`) to output schema — checked first, heuristic fallback
2. Add `enum` field type with `enumValues: string[]` — enables constrained verdicts and proper rendering
3. Remove `max_score = 100` default — if no thresholds, max is `None`
4. Cross-run aggregation is out of scope for now but documented as future work
