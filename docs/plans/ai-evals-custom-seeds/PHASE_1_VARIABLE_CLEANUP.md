# Phase 1: Variable Registry Cleanup

**Goal:** Remove redundant and pipeline-internal variables from the registry and resolver. Rename remaining variables for clarity. Expand dynamic path extraction to cover the full API response.
**Risk Level:** Low — removing dead code, renaming display labels, one resolver behavior change (path extraction scope).
**Prerequisite:** None.

---

## 1. Variable Registry Changes

**File:** `backend/app/services/evaluators/variable_registry.py`

### Variables to REMOVE (delete entirely from `_register_voice_rx_variables`):

| Key | Why Remove |
|---|---|
| `api_input` | Redundant — `{{input}}` resolves via dynamic path to `api_response["input"]` |
| `structured_output` | Redundant — `{{rx}}` resolves via dynamic path to `api_response["rx"]` |
| `api_rx` | Replaced by `api_response` (rename) |
| `script_preference` | Pipeline-internal. Resolves to meaningless default ("roman") in custom evaluators because the runner doesn't pass `prerequisites` context. Used only by standard eval pipeline internally. |
| `language_hint` | Pipeline-internal. Same reason — always "Not specified" in custom evals. |
| `preserve_code_switching` | Pipeline-internal. Always "yes" in custom evals. |
| `original_script` | Pipeline-internal. Always "auto" in custom evals. |

### Variables to RENAME (key + display name + category):

| Old Key | New Key | New Display Name | New Category | New Description |
|---|---|---|---|---|
| `audio` | `audio` *(no change)* | Audio Recording | `input` | The audio recording. Triggers audio-capable LLM call. |
| `transcript` | `transcript` *(no change)* | Uploaded Transcript | `input` | Full transcript as [Speaker]: text lines from the uploaded listing data. |

| `rx.*` | `rx.*` *(no change)* | API Fields (dot-path) | `api_data` | Access any field in the API response via dot notation. Examples: `{{input}}` for transcript, `{{rx}}` for structured data, `{{rx.vitals.temperature}}` for specific values. |

## 3. Dynamic Path Extraction Change

**File:** `backend/app/routes/evaluators.py`

### Change `list_api_paths` endpoint (lines 112-121):

**BEFORE:**
```python
@router.get("/variables/api-paths")
async def list_api_paths(
    listing_id: str = Query(..., alias="listingId"),
    db: AsyncSession = Depends(get_db),
):
    """Extract available {{rx.*}} variable paths from a listing's API response."""
    listing = await db.get(Listing, listing_id)
    if not listing or not listing.api_response:
        return []
    return _extract_paths(listing.api_response, prefix="rx")
```

**AFTER:**
```python
@router.get("/variables/api-paths")
async def list_api_paths(
    listing_id: str = Query(..., alias="listingId"),
    db: AsyncSession = Depends(get_db),
):
    """Extract available variable paths from a listing's API response."""
    listing = await db.get(Listing, listing_id)
    if not listing or not listing.api_response:
        return []
    return _extract_paths(listing.api_response, prefix="")
```

The `_extract_paths` function already handles empty prefix (line 26: `f"{path}.{k}" if path else k`). With `prefix=""`, it produces paths like `input`, `rx`, `rx.vitals`, `rx.vitals.temperature` — all of which resolve correctly via `_get_nested_value(api_response, path)` in the dynamic fallback.

### Update VariablePickerPopover section header:

The frontend `VariablePickerPopover.tsx` (line 193) currently shows the header "API RESPONSE DATA" for the dynamic paths section. This is already accurate. No frontend change needed.

## 4. Frontend Changes

**None required.** The variable picker renders categories and variables dynamically from the backend. All changes are backend-only:
- Categories come from the `category` field on each variable
- Variables are fetched from `/api/evaluators/variables`

The picker will automatically reflect the new categories and cleaned-up variable list after the backend changes.
